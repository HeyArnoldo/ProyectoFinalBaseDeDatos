\section{Privacidad y seguridad de los datos}

El pedido contiene nombre, teléfono y dirección, datos que deben tratarse como información personal. El conjunto semilla y las evidencias usan identidades completamente ficticias. No se incorporan credenciales, tokens, teléfonos reales ni direcciones reales al repositorio o al informe. Las capturas se revisan antes de su inclusión y la API limita las lecturas operacionales al operador autenticado.

La autenticación implementada es proporcional a un prototipo local con un único operador, pero no equivale a un sistema completo de gestión de identidades. Un despliegue real requeriría rotación de secretos, protección CSRF evaluada según el mecanismo de cookie, auditoría, recuperación de cuenta, cifrado en tránsito y políticas de conservación de datos más completas \parencite{owaspASVS}.

\section{Limitaciones y validez}

\subsection{Limitaciones técnicas}

\begin{itemize}
  \item MongoDB y Cassandra se ejecutan con un nodo cada uno; no se demuestra tolerancia a fallos de nodo.
  \item El trabajador está embebido en una única API; no se evalúa coordinación entre múltiples trabajadores.
  \item Las dos tablas Cassandra soportan solo las consultas declaradas; nuevas consultas exigirían nuevas proyecciones.
  \item No se incluyen pagos, mensajería ni despliegue productivo.
  \item La memoria limitada favorece reproducibilidad local, no una comparación de rendimiento entre motores.
\end{itemize}

\subsection{Amenazas a la validez}

La \textbf{validez interna} puede verse afectada si quedan datos de ejecuciones anteriores; por ello se registra el estado de los volúmenes y se usan claves fijas. La \textbf{validez de constructo} exige que «recuperación» no se reduzca a que el contenedor vuelva a iniciar: se prueba la convergencia de eventos. La \textbf{validez externa} es limitada, pues un entorno de nodo único y carga pequeña no representa producción. La \textbf{confiabilidad} se fortalece ejecutando cada evidencia al menos dos veces desde la misma semilla y conservando salidas comparables.

\section{Plan de reversión}

Si una versión del trabajador produce proyecciones incorrectas, se detendrá su ejecución sin modificar MongoDB. Las tablas Cassandra podrán eliminarse y reconstruirse a partir de los eventos de salida conservados después de corregir el defecto. Si falla una versión de la API o de la interfaz, se restaurará la imagen anterior y se preservarán los volúmenes. Toda operación destructiva deberá estar precedida por una copia de la evidencia y una verificación de que MongoDB mantiene el historial necesario para la reproducción.
